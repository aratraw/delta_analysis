SURVIVAL INSTRUCTION FOR RATIONAL CORE: A DEEP DIVE INTO 4 STRAIGHT DAYS OF DEBUGGING
================================================================================

1. ARCHITECTURAL PRINCIPLES
---------------------------
1.1 Node Pool
   - Use std::vector<Node> and std::vector<Value> for contiguous memory and cache locality.
   - Use int32_t indices instead of pointers (children, references).
   - Use inline thread_local NodePool pool for thread-isolated, lock-free storage.

1.2 Hash Consing (Interning)
   - Maintain three caches:
       * value_cache   : Value → index (for constants)
       * binary_cache  : (op, left, right) → index (for ADD, MUL)
       * unary_cache   : (op, child, value_idx) → index (for NEG, RECIP, transcendental)
   - Always canonicalize keys before lookup (normalize numbers, order operands).

1.3 Hashing
   - Hash must be consistent with equality.
   - For Value: normalize via normalize_to_cpp_int(v) → hash (num, den) as cpp_int.
   - For Node: combine hash of children, operation, value_idx.
   - Invariant: if a == b, then hash(a) == hash(b) — critical for absl containers.

1.4 Canonicalization
   - Numbers: normalized on construction (den > 0, fraction reduced).
   - Binary ADD/MUL: children ordered by hash.
   - SUB and DIV: transform to ADD/MUL with NEG/RECIP.

2. API DESIGN
-------------
2.1 Rational Constructors (Immediate)
   - Rational() = default;
   - Rational(absl::int128 num);
   - Rational(absl::int128 num, absl::uint128 den);
   - explicit Rational(const cpp_int& num);
   - Rational(const cpp_int& num, const cpp_int& den);
   - explicit Rational(const std::string& s);
   - Rational(int num, int den);
   - Explicit constructors for int, long long, unsigned long long (to intercept literals).
   - Delete float/double constructors.

2.2 Lazy Creation (no ambiguous constructor)
   - Use static factory: static Rational from_lazy_index(std::size_t root_idx);
   - Never use a constructor from size_t.

2.3 State Queries
   - bool is_immediate() const noexcept;
   - bool is_lazy() const noexcept;

2.4 Accessors
   - const SmallStorage* as_small() const noexcept;
   - const BigStorage* as_big() const noexcept;
   - int root_index() const;  // requires is_lazy()

2.5 Operators
   - Implement +, -, *, /, unary - in rational_impl.h.
   - Rule: if both immediate → eager_*, else build lazy tree.
   - Use Rational::from_lazy_index to create lazy results, never direct constructor.

3. ALGORITHMS
-------------
3.1 Building Expressions
   - ExpressionRoot is a thin wrapper over int (root index).
   - Factories: make_const, make_unary, make_binary.
   - Arithmetic: add, mul, sub (via add(neg(...))), div (via mul(recip(...))).
   - Transcendentals: sqrt, exp, log, sin, cos, acos with eps.

3.2 Evaluation (eval)
   - Iterative post‑order with explicit stack.
   - Local cache: std::vector<std::optional<Value>> cache(n).
   - Special case for leaf-less nodes (PI, E) — compute directly.

3.3 Simplification (simplify)
   - Iterative post‑order with cache of simplified indices.
   - Rules:
       * Constant folding: compute through eager.
       * Unary: NEG(NEG(x)), RECIP(RECIP(x)), EXP(LOG(x)), LOG(EXP(x)).
       * Binary: x+0, x*1, x*0, x+(-x), x*(1/x).

4. MEMORY MANAGEMENT
--------------------
4.1 Pool Growth
   - Nodes are never deleted; pool only grows. Acceptable for up to thousands of nodes.
   - For long-running apps: implement garbage collection (copy reachable nodes to new pool).

4.2 Thread Safety
   - inline thread_local NodePool pool → each thread has its own, no locks.

5. ERROR HANDLING
-----------------
5.1 Domain Errors
   - Division by zero → std::domain_error.
   - sqrt of negative, log of non-positive → std::domain_error.

5.2 Hash Invariants
   - Must hold: if a == b then hash(a) == hash(b). Violations cause assertion failures in absl debug builds.

6. KEY LESSONS LEARNED
----------------------
1. Constructors are a source of ambiguity. Use factory methods when multiple creation paths exist.
2. Hashing and equality must be a unified system. Always normalize before hashing if objects can be equal in non-normalized form.
3. Prefer iterative traversals over recursion. Stack depth is limited, expression depth can be arbitrary.
4. Thread-local pools are simple and lock-free.
5. Canonicalize at every level: numbers, nodes, cache keys.
6. Tests are a mirror of architecture. If a test expects a specific child order, the API is fragile.

7. FINAL CODE STRUCTURE (≈3000 lines)
-------------------------------------

This codebase now serves as a reference implementation for lazy rational arithmetic with:

CONCRETE LESSONS LEARNED (THE BLOODY DETAILS)
=============================================

LESSON 1: NEVER, EVER, EVER USE A CONSTRUCTOR FROM std::size_t FOR LAZY OBJECTS
--------------------------------------------------------------------------------
The Disaster:
    Rational(int root_idx) // constructor from int
    vs
    Rational(std::size_t root_idx) // lazy constructor

What Happened:
    Rational(0)   // ambiguous: could be immediate zero or lazy index 0
    Rational(1)   // ambiguous: could be immediate one or lazy index 1
    The compiler picked the lazy constructor because size_t was "more special"
    → immediate zero became a lazy expression → vector subscript out of range

The Fix:
    Delete all single-argument constructors that accept integer types.
    Use static factory: static Rational from_lazy_index(std::size_t root_idx).

Golden Rule:
    If a type can be constructed in fundamentally different ways (immediate value vs lazy reference),
    use a factory method for the less common case. Constructors are for "normal" construction.

LESSON 2: HASH FUNCTIONS MUST BE PERFECTLY CONSISTENT WITH EQUALITY
-------------------------------------------------------------------
The Disaster:
    Value can be SmallStorage(2,4) or BigStorage(1,2) after normalization.
    operator== normalizes both to (1,2) and says they are equal.
    hash(2,4) != hash(1,2) → absl flat_hash_map fires assertion.

The Fix:
    template <typename H>
    H AbslHashValue(H h, const Value& v) {
        auto [num, den] = normalize_to_cpp_int(v); // always canonical representation
        return H::combine(std::move(h), num, den);
    }

Golden Rule:
    Hash must be computed from the canonical representation of the object,
    not from the raw storage. Otherwise, equal objects will have different hashes,
    breaking hash-based containers.

LESSON 3: NORMALIZATION IS NOT OPTIONAL FOR VALUES IN CACHES
------------------------------------------------------------
The Disaster:
    SmallStorage(2,4) with reduced=false was inserted into value_cache.
    Later, SmallStorage(1,2) with reduced=true was compared.
    operator== normalized, but hash on raw storage → mismatch.

The Fix:
    Every Value entering the cache must be normalized:
    - In add_value(), normalize before lookup and insertion.
    - In eager_* functions, normalize results before returning.

Golden Rule:
    If an object has a "reduced" flag, never trust it when it enters a global cache.
    Normalize at the entry point, always.

LESSON 4: NEVER NORMALIZE INSIDE HASH FUNCTION WITHOUT NORMALIZING THE STORED VALUE
------------------------------------------------------------------------------------
The Disaster:
    AbslHashValue normalized on the fly, but the stored Value remained non-normalized.
    Next lookup: hash(normalized copy) ≠ hash(stored raw) → not found → duplicate inserted.

The Fix:
    Normalize at insertion time, store normalized. Hash function can then assume normalized.

Golden Rule:
    Normalize once at the point of insertion, not on every hash call.
    Otherwise, you create a "phantom key" that never matches.

LESSON 5: WHEN YOU CHANGE THE API, GREP EVERYTHING, THEN GREP AGAIN
--------------------------------------------------------------------
The Disaster:
    Changed Rational(std::size_t) → Rational::from_lazy_index.
    Updated rational_impl.h, but forgot transcendentals.h and lazy_test.cpp.
    Result: some calls still used Rational(static_cast<size_t>(...)) → ambiguous.

The Fix:
    After API change: grep -r "Rational(static_cast<size_t>" and kill all occurrences.
    Then compile, fix errors, grep again.

Golden Rule:
    API changes are like nuclear waste: they contaminate everything. Use grep, sed, and static analysis.

LESSON 6: IMMEDIATE AND LAZY MUST BE TOTALLY SEPARATED IN THE TYPE SYSTEM
--------------------------------------------------------------------------
The Disaster:
    Same type (Rational) could be immediate or lazy, but constructors made it ambiguous.
    There was no way to tell the compiler "I want an immediate zero" vs "I want a lazy index 0".

The Fix:
    Immediate: Rational(absl::int128(0)) — explicit conversion.
    Lazy: Rational::from_lazy_index(0) — factory.

Golden Rule:
    If a type has two fundamentally different modes of existence,
    either use different types (immediate_rational vs lazy_rational)
    or use factory methods to disambiguate. Never rely on implicit conversions.

LESSON 7: RECURSION IS THE ENEMY OF DEEP TREES
-----------------------------------------------
The Disaster:
    Recursive evaluate() could overflow stack on deep chains (1000+ nodes).

The Fix:
    Iterative post‑order traversal with explicit stack and local cache.

Golden Rule:
    In lazy evaluation, expression depth is arbitrary. Never recurse.
    Always use iterative algorithms with explicit data structures.

LESSON 8: THREAD-LOCAL IS CHEAPER THAN LOCKS
---------------------------------------------
The Disaster:
    Global pool with mutex would kill performance in multi-threaded code.

The Fix:
    inline thread_local NodePool pool — each thread has its own independent pool.

Golden Rule:
    If data does not need to be shared across threads, use thread_local.
    It's lock-free, cache-local, and simpler than mutexes.

LESSON 9: TESTS THAT EXPECT A SPECIFIC ORDER ARE BRITTLE
---------------------------------------------------------
The Disaster:
    test LazyTreeStructure expected left child = ADD, right child = CONST.
    After canonicalization (operands sorted), order could swap → test failed.

The Fix:
    Test should accept both orders: one child is ADD, the other is CONST.

Golden Rule:
    Tests should verify invariants, not implementation details.
    If the implementation may change order, the test must be order-agnostic.

LESSON 10: DEBUGGING HASH MAPS IS A NIGHTMARE, SO GET THE HASH RIGHT
---------------------------------------------------------------------
The Disaster:
    absl::flat_hash_map would crash with "hash/eq functors are inconsistent"
    after hours of debugging, because Value::operator== and Value::hash disagreed.

The Fix:
    Make operator== and hash both work on canonical representation.
    Test with asserts in debug mode: assert((a == b) == (hash(a) == hash(b))).

Golden Rule:
    Hash-based containers are unforgiving. Before using them, ensure that
    the hash function is a true mathematical function consistent with equality.
    Test this invariant in debug builds.

The Fix:

Golden Rule:

Rational Library Mode Behavior (Final Specification)
====================================================

1. DEFAULT BEHAVIOR – IMMEDIATE
--------------------------------
All constructors create immediate objects by default.
- Rational a = 1_r;        → immediate
- Rational b = "1/2"_r;    → immediate
- Rational c(1, 2);        → immediate

This ensures maximum performance for hot loops, mesh calculations,
and intermediate computations where lazy evaluation is not required.

2. ARITHMETIC OF TWO IMMEDIATE NUMBERS → IMMEDIATE
---------------------------------------------------
If both operands are immediate, the result is also immediate:
- a + b, a - b, a * b, a / b  → immediate
- No nodes are created in the pool.
- No lazy tree overhead.

This is the fast path optimized for performance-critical code.

3. LAZY EVALUATION – EXPLICIT ONLY
-----------------------------------
Lazy trees are built only when explicitly requested:
- .lazy() converts an immediate into a constant node
- If at least one operand is lazy, operators build a lazy tree
- Example: a.lazy() + b  → lazy tree (even if b is immediate)

This gives the programmer full control over where lazy evaluation is used.

4. ScopedEagerEval – TEMPORARY FORCED EAGER MODE
-------------------------------------------------
ScopedEagerEval temporarily forces immediate evaluation for all operations
within the block, regardless of operand types.
- Useful for numerical verification, debugging, or quick approximate results
- After exiting the block, the previous mode is restored

Example:
    set_eager_mode(false);  // normal behavior
    {
        ScopedEagerEval guard;  // inside: forced eager mode
        Rational sum = a.lazy() + b;  // evaluated immediately
        EXPECT_FALSE(sum.is_lazy());
    }
    // After block: normal behavior restored

5. TRANSCENDENTAL FUNCTIONS
---------------------------
- In eager mode (global_eager_mode == true or ScopedEagerEval active):
  The function is evaluated immediately using fast/slow path based on epsilon.
- In lazy mode (global_eager_mode == false and not inside ScopedEagerEval):
  A lazy tree node is created (SQRT, EXP, LOG, SIN, COS, ACOS, PI, E).

6. SUMMARY OF MODES
-------------------
| Scenario                                    | Result        |
|---------------------------------------------|---------------|
| Both operands immediate (normal mode)       | immediate     |
| At least one operand lazy (normal mode)     | lazy          |
| Inside ScopedEagerEval block                | immediate     |
| global_eager_mode = true                    | immediate     |

This design ensures:
- Optimal performance by default
- Explicit control over lazy evaluation
- Easy numerical verification via ScopedEagerEval
- Predictable, documented behavior

NEW LESSONS FROM THE TRENCHES (ADDED AFTER THE GREAT CUBE ROOT DEBACLE)
========================================================================

LESSON 13: FAST PATH IS NOT ALWAYS ACCURATE – KNOW YOUR OPERATION'S CONDITION NUMBER
-------------------------------------------------------------------------------------
The Disaster:
    For eps = 1e-30, fast_nth_root using cpp_dec_float_100 gave insufficient precision
    for cube root (n=3). 1/3 is not exactly representable in binary, and pow(fx, 1/n)
    accumulates error. The result had error ~1e-19, while required was 1e-28.

The Fix:
    - Use fast path (through cpp_dec_float_100) ONLY for square root (n=2), where 1/2 is exact.
    - For n>2, always use Newton's method with rational arithmetic, regardless of eps.

Golden Rule:
    Never assume that a floating-point approximation of a transcendental operation
    meets a user-specified epsilon without rigorous error analysis. When in doubt,
    use the exact (slow) path. Speed is worthless if the answer is wrong.

LESSON 14: INFINITE SERIES MUST NOT BE CUT OFF BY AN ARBITRARY ITERATION LIMIT
-------------------------------------------------------------------------------
The Disaster:
    slow_exp, slow_log, slow_sin, slow_cos, slow_pi had const size_t max_iter = 10000.
    For eps = 1e-30 and certain arguments, convergence may require >10000 terms.
    The loop would stop by limit, not by precision, giving wrong result.

The Fix:
    Remove max_iter entirely, rely solely on is_less(term, eps). Add a safety guard only
    to detect divergence, not to limit convergence.

Golden Rule:
    A series that converges to a limit must be allowed to run until the desired precision
    is achieved, not until a hardcoded iteration count. Respect the user's epsilon.

LESSON 15: DO NOT HARDCODE EXPECTED VALUES IN TESTS FOR TRANSCENDENTAL FUNCTIONS
---------------------------------------------------------------------------------
The Disaster:
    Test EagerPowHighPrecision used a hardcoded Rational constant for 5^(1/3) with only
    19 decimal digits of precision. The actual result had 1e-19 error, but required
    tolerance was 1e-28. The test failed because the "expected" value was not accurate enough.

The Fix:
    Compute the expected value on the fly inside the test using a high‑precision
    floating‑point library (boost::multiprecision::cpp_dec_float_100) with 100 digits,
    then convert to Rational via string. This guarantees the reference is more accurate
    than the tolerance.

Golden Rule:
    For tests of transcendental functions, never hardcode decimal approximations.
    Either compute the reference using a trusted high‑precision library, or use
    relative tolerances with a precomputed high‑accuracy value (100+ digits).

LESSON 16: FUNCTIONS MUST RESPECT THE REQUESTED EPSILON – NOT BE TOO PRECISE, NOT TOO ROUGH
-------------------------------------------------------------------------------------------
The Disaster:
    In some earlier function (before this chat), we added an extra safety margin of 5-10 digits
    when converting from high‑precision float to Rational. This caused a test like:
        Rational high = sqrt(2, 1e-30);
        Rational low  = sqrt(2, 1e-2);
        EXPECT_GT(abs(high - low), 1e-6);
    to fail because low was computed with far higher precision than requested (due to extra digits).
    The test expected a difference of at least 1e-6, but got a difference of ~1e-30 because
    both calls returned essentially the same high‑precision result.

The Fix:
    Remove any artificial extra precision in the fast path. The parameter eps must be the
    target error bound, not a lower bound. If the user asks for 1e-2, we must compute with
    error ≈ 1e-2, not 1e-30. Extra digits may be added only for internal guarding, but the
    final result must be rounded/truncated to the requested tolerance.

Golden Rule:
    A function that accepts an epsilon parameter must return a result whose error is
    *at most* epsilon, but also not *much less* than epsilon if that would violate
    the expected behavior (e.g., when testing that lower precision is faster or yields
    a different value). Always respect the user's precision request.

LESSON 17: HYBRID THRESHOLD MUST BE ADJUSTED PER OPERATION, NOT GLOBAL
-----------------------------------------------------------------------
The Disaster:
    HYBRID_THRESHOLD = 1e-35 was chosen globally. For sqrt, fast path with cpp_dec_float_100
    is accurate even for eps=1e-30. For cbrt, it is not. A single threshold cannot serve
    all operations.

The Fix:
    Make the decision per operation:
        - sqrt: use fast path if to_double(eps) >= 1e-35 (safe)
        - nth_root (n>2): never use fast path (or only for very large eps like 1e-10)
        - exp/log/sin/cos: fast path with cpp_dec_float_100 is generally safe.

Golden Rule:
    A global threshold is a code smell. Each transcendental function should have its own
    heuristic for switching between fast approximate and slow exact paths.

LESSON 18: to_double() ON VERY SMALL RATIONALS CAN BE DENORMAL OR ZERO
-----------------------------------------------------------------------
The Disaster:
    compute_extra_digits calls to_double(eps) to compute log10. For eps = 1e-30,
    to_double returns ~1e-30 (subnormal) – still fine. But for eps = 1e-100,
    to_double may underflow to 0.0, causing log10(0) → -inf and extra_digits nonsense.

The Fix:
    In compute_extra_digits, if eps_double <= 0, return a large safety margin (e.g., 100).
    Also, avoid relying on to_double for very small eps; use cpp_dec_float_100 to compute
    digits_needed directly.

Golden Rule:
    Converting arbitrary‑precision rationals to double is lossy and can underflow/overflow.
    Use high‑precision arithmetic for decisions that depend on extremely small values.

SUMMARY OF THE LATEST BLOODY DETAILS
====================================
- Fast path is not universally accurate; square root is safe, cube root is not.
- Infinite series must not be limited by arbitrary iteration counts; respect epsilon.
- Test expectations must be computed with higher precision than the tolerance.
- Functions must respect the requested epsilon – not too rough, not too precise.
- Hybrid thresholds must be per operation.
- Double conversion can underflow for tiny eps; use high‑precision arithmetic.
- Hardcoded decimals in tests are evil.

These lessons cost another day of debugging but finally made the library robust
for the full range of eps from 1e-35 to 1e-30 and beyond.

LESSON 19: WHEN COMPOSING FUNCTIONS, PASS A STRICTER EPSILON TO SUBCOMPUTATIONS
-------------------------------------------------------------------------------
The Disaster:
    In eager_pow(base, exp, eps) for rational exponent p/q, we need to compute:
        result = exp((p/q) * log(base))
    If we call eager_log(base, eps) and eager_exp(..., eps), the error from log
    and exp can accumulate. The final error may exceed eps even if each individual
    function respects eps.

The Fix:
    Always pass a stricter epsilon to subcomputations:
        internal_eps = eps / (p * 1000)   (safety factor 1000)
    Similarly, for nth_root, we call internal functions with eps/1000.

Golden Rule:
    When a function F(x, eps) calls other transcendental functions G(y, eps_internal),
    eps_internal must be significantly smaller than eps (e.g., eps / 1000 or eps / (p*1000)).
    The safety factor should account for:
        - Number of composed functions
        - Magnitude of intermediate values (amplification factors)
        - Desired final error bound
    Never pass the same eps to subcomputations, or the final error will be ~ n * eps,
    potentially exceeding the user's tolerance.

Example from our code:
    Value internal_eps = eager_div(eps, SmallStorage(1000));
    Value log_base = eager_log(base, internal_eps);
    Value result = eager_exp(p_log_div_q, internal_eps);

This ensures that the final error is dominated by the outermost approximation,
not by the accumulation of inner errors.

Consequence for users:
    The library may be slightly slower when computing composed functions because
    it asks for higher precision internally. This is intentional and correct.

LESSON 20: WHEN IN DOUBT – USE COUT (AND TO_STRING, NEVER TO_DOUBLE)
---------------------------------------------------------------------
The Disaster:
    For hours (or days), we stared at failing tests for pow(5, 1/3) with eps=1e-30.
    We tried to infer from error messages, from EXPECT_LE output, from to_double()
    conversions. Nothing made sense because to_double() hid the true precision
    (both values printed the same in double: 1.7099759466766970561).

The Fix:
    Finally, we added:
        std::cout << "val = " << delta::to_string(val) << "\n";
        std::cout << "expected = " << delta::to_string(expected) << "\n";
        std::cout << "diff = " << delta::to_string(diff) << "\n";
    And immediately saw the truth: expected had only 19 decimal digits,
    while val had hundreds. The test was comparing a high‑precision result
    against a low‑precision hardcoded constant.

Golden Rule:
    When debugging numerical code, never trust:
        - Debugger watch windows (they often truncate)
        - Implicit conversions to double (they lose precision)
        - EXPECT_EQ / EXPECT_LE output from test frameworks (they show truncated hex dumps)
    Instead:
        - Print full rational values using to_string()
        - Print with high precision: std::setprecision(20) if using double
        - For containers, dump key elements with full precision
    cout (or any stream) with to_string() is your magnifying glass.
    Use it liberally, then remove or guard with #ifdef DEBUG.

Corollary:
    If a test fails and you cannot see why, the first step is not to change the algorithm.
    The first step is to print the actual values in their full glory.
    You cannot fix what you cannot see.
