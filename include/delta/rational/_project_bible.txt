# Техническое описание архитектуры ленивых вычислений `delta::Rational` (актуальная версия 2.0)

## 0. Цели и объём

### 0.1. Основные цели

1. **Максимально ускорить построение деревьев ленивых выражений** – отказ от любых упрощений, хэширования, интервалов и нормализации на этапе накопления.
2. **Ввести N‑арные узлы `SUM` и `PRODUCT`** вместо бинарных `ADD`/`MUL`.
3. **Разделить типы** – `Rational` (всегда immediate) и `LazyRational` (всегда ленивый).
4. **Мутирующие операторы** для `LazyRational` – накопление без копирования.
5. **Однократная канонизация** – вся тяжёлая работа (нормализация, интернирование, упрощение) выполняется один раз по требованию.
6. **Упростить дерево операций** – нет узлов `SUB` и `DIV` (выражаются через `NEG`/`RECIP`).

### 0.2. Что не входит в рефакторинг

- Трансцендентные функции (`sqrt`, `exp`, `log`, `sin`, `cos`, `acos`, `pow`) остаются унарными/бинарными.
- Глобальный пул чистых узлов (`NodePool`) не переписывается кардинально, только адаптируется.
- Алгебраические упрощения (дистрибутивность, приведение подобных) не являются частью обязательной канонизации (могут быть добавлены позже).

## 1. Архитектурные принципы и инварианты

### 1.1. Разделение типов

- **`Rational`** – всегда immediate число. Арифметика между двумя `Rational` выполняется **eager** и возвращает новый `Rational`.  
  **Инвариант:** `Rational` никогда не содержит ленивого состояния.
- **`LazyRational`** – всегда ленивое выражение. Может находиться в двух состояниях:  
  *Dirty* (грязное, мутабельное дерево) или *Clean* (чистое, интернированное в глобальном пуле).  
  **Инвариант:** `LazyRational` не может быть immediate.

### 1.2. Грязное дерево (Dirty)

- Принадлежит **одному** объекту `LazyRational` (move‑only, копирование запрещено).
- Хранится в виде прямых полей (без `unique_ptr`):
  - `std::vector<internal::DirtyNode> nodes_` – узлы, индексы от 0 до size-1.
  - `std::vector<internal::Value> constants_` – локальный пул констант (без нормализации).
  - `int root_` – индекс корневого узла.
- **Инварианты Dirty-дерева:**
  - Нет глобальных кэшей, нет хэшей, нет интервалов, нет счётчиков ссылок.
  - Узлы ссылаются на детей по индексам в `nodes_`.
  - `CONST` хранит `const_index` в `constants_`.
  - `SUM`/`PRODUCT` – дети в хронологическом порядке (не сортированы).
  - Унарные операции (`NEG`, `RECIP`, `SQRT`, …) – `children.size() == 1`.
  - `POW` – `children.size() == 2`.
  - Узлов `SUB` и `DIV` не существует: `a - b` → `a + NEG(b)`, `a / b` → `a * RECIP(b)`.
- **Мутации**: только если объект в состоянии Dirty; если Clean, сначала вызывается `ensure_dirty()`, который копирует чистое дерево в грязное.

### 1.3. Чистое дерево (Clean)

- Представлено глобальным пулом `NodePool` (один экземпляр на поток, thread‑local).
- Узлы имеют хэши, интервалы, `refcount`, кэшируются и интернируются.
- `SUM` и `PRODUCT` – N‑арные, **канонические**:
  - дети отсортированы по хэшу (для детерминизма),
  - нет нейтральных элементов (0 для `SUM`, 1 для `PRODUCT`),
  - **константы НЕ сворачиваются** – это ответственность `evaluate`, который выполняет суммирование эффективно (последовательное сложение с нормализацией).
- Узлы `ADD`, `MUL`, `SUB`, `DIV` отсутствуют.

### 1.4. Кэш интервалов в `LazyRational`

- `mutable std::optional<internal::Interval> cached_interval_`.
- Вычисляется лениво при вызове `approx_interval()`.
- Сбрасывается (`reset()`) при любой мутации (операторы `+`, `-`, `*`, `/`, `+=` и т.д., а также `ensure_dirty()`).
- `approx_interval()` **не вызывает канонизацию** – интервал вычисляется непосредственно от грязного дерева (итеративно, без создания чистых узлов) и кэшируется. Это позволяет получать приблизительную оценку без затрат на упрощение.

### 1.5. Канонизация (Dirty → Clean) – трёхфазный процесс

Вызывается только по требованию: `simplify_inplace()`, а также внутри `eval()` (если `skip_simplify == false`) и операторов сравнения.

**Фаза 1: Построение локального чистого дерева (`TempNode`)**  
- Итеративный пост-порядный обход грязного дерева.
- Для каждого грязного узла создаётся `TempNode`:
  - вычисляется `hash`, `approx`, `depth` (аналогично глобальному пулу, но без кэширования),
  - константы копируются в локальный `temp_values`.
- Результат – временное дерево, синтаксически эквивалентное грязному.

**Фаза 2: Локальное упрощение (`simplify_tree`)**  
- Выполняется над `TempNode` (без глобального пула).
- Включает:
  - flattening (расплющивание вложенных `SUM`/`PRODUCT`),
  - удаление нейтральных элементов (0 для `SUM`, 1 для `PRODUCT`),
  - сортировку детей по хэшу,
  - **НЕ сворачивает константы** – только алгебраические упрощения,
  - алгебраические упрощения: `NEG(NEG(x))→x`, `RECIP(RECIP(x))→x`, `EXP(LOG(x))→x`, `LOG(EXP(x))→x`,
  - сокращение пар в `SUM`: `x + NEG(x) → 0`,
  - сокращение пар в `PRODUCT`: `x * RECIP(x) → 1`,
  - упрощения для `POW`: `x^0→1`, `x^1→x`, `1^x→1`, `0^positive→0`, `(a^b)^c→a^(b*c)` для целых показателей.
- Возвращает индекс упрощённого корня в том же массиве `TempNode`.

**Фаза 3: Интернирование в глобальный пул**  
- Обход упрощённого дерева `TempNode` снизу вверх.
- Для каждого узла вызывается соответствующая «глупая» фабрика пула:
  - `add_const` – нормализует и кэширует константу,
  - `make_sum_node` / `make_product_node` – создаёт узел с переданными детьми (без сортировки, фильтрации, свёртки),
  - `get_unary_node`, `get_pow_node` – аналогично.
- Полученные глобальные индексы сохраняются в `temp_to_global`.
- Корень заменяется на упрощённый и интернированный.
- Увеличивается `refcount` нового чистого узла.

После канонизации:
- `state_ = Clean`,
- `clean_index_ = new_root`,
- грязные векторы (`nodes_`, `constants_`, `root_`) очищаются,
- кэш интервала сбрасывается (хотя он уже не нужен).

### 1.6. Мутирующие операторы

- Все бинарные операторы `+`, `-`, `*`, `/` для `LazyRational` **мутируют левый операнд** (lvalue или rvalue).
- Возвращают ссылку на левый операнд (`LazyRational&` или `LazyRational&&`).
- Перед мутацией вызывают `ensure_dirty()` и `invalidate_interval()`.
- Реализация:
  - Если корень левого операнда не `SUM` (для `+`) – создаётся новый узел `SUM`, оборачивающий старый корень и правый операнд (копируемый через `import_tree`).
  - Если корень уже `SUM` – правый операнд добавляется в список детей (с расплющиванием, если правый тоже `SUM`).
- Вычитание и деление транслируются в `+ NEG` и `* RECIP` соответственно.
- Составные операторы `+=`, `-=`, `*=`, `/=` делегируют бинарным.

### 1.7. Вычисление (`eval` и `eval_inplace`)

- **`Rational eval(bool skip_simplify = false) const`**  
  - Если `skip_simplify == false` (по умолчанию) – канонизирует объект (если Dirty) и вычисляет через `internal::evaluate`.  
  - Если `skip_simplify == true` и объект Dirty – вычисляет **напрямую** по грязному дереву (через `eval_dirty()`), без канонизации.  
  - **Не изменяет объект** (если вычисляется грязное дерево, создаётся временный кэш, но сам объект остаётся Dirty).  

- **`void eval_inplace(bool skip_simplify = false)`**  
  - Аналогично, но результат заменяет текущее дерево на константу.  
  - Если `skip_simplify == true` и объект Dirty – вычисляет через `eval_dirty()` и заменяет на константу, минуя канонизацию.  

- **Внутренняя оптимизация** – `evaluate` для узлов `SUM` использует **последовательное попарное сложение** с нормализацией после каждого шага (вместо глобального LCM), что предотвращает взрывной рост знаменателя и обеспечивает линейное время для больших сумм.

### 1.8. Сравнения (многоуровневая логика)

Операторы сравнения для `LazyRational` реализуют трёхступенчатую эвристику для минимизации дорогих вычислений:

1. **Быстрая проверка** – если оба объекта чистые и их индексы равны → сразу `true`/`false`.
2. **Интервальная проверка** – вычисляются приблизительные интервалы (через `approx_interval()`). Если интервалы не пересекаются, можно сразу дать ответ без канонизации.
3. **Канонизация** – если интервалы пересекаются, оба объекта канонизируются (становятся чистыми).
4. **Сравнение индексов** – после канонизации, если индексы равны – `true`.
5. **Полное вычисление** – в противном случае вызывается `eval()` для обоих операндов и сравниваются значения.

Этот подход значительно ускоряет сравнения в случаях, когда значения заведомо различны (непересекающиеся интервалы) или одинаковы (один и тот же чистый узел).

### 1.9. Глобальный пул (`NodePool`) – «глупые» фабрики

- Фабричные функции (`make_sum_node`, `make_product_node`, `get_unary_node`, `get_pow_node`) **не выполняют никаких упрощений**.
- Они только:
  - проверяют кэш (по исходным детям, в том же порядке),
  - вычисляют `depth`, `approx`, `hash` на основе переданных детей,
  - выделяют узел и добавляют в кэш.
- Упрощения вынесены в `simplify_tree`, который вызывается до интернирования.

### 1.10. Итеративные обходы

- Все обходы деревьев (построение `TempNode`, `simplify_tree`, интернирование, вычисление интервала от грязного дерева, `import_tree`) реализованы с использованием явного стека (`std::stack`).
- Рекурсия отсутствует, глубина дерева может быть произвольной.

## 2. Детальное описание интерфейсов

### 2.1. Класс `Rational` (immediate)

(Без изменений относительно предыдущей версии. Ключевые методы: конструкторы, арифметика, сравнения, `as_lazy()`.)

### 2.2. Класс `LazyRational`

```cpp
class LazyRational {
public:
    LazyRational();                              // CONST(0)
    explicit LazyRational(const Rational& r);
    explicit LazyRational(Rational&& r);

    // move-only
    LazyRational(const LazyRational&) = delete;
    LazyRational& operator=(const LazyRational&) = delete;
    LazyRational(LazyRational&&) noexcept;
    LazyRational& operator=(LazyRational&&) noexcept;
    ~LazyRational();

    LazyRational clone() const;                  // глубокое копирование

    Rational eval(bool skip_simplify = false) const;   // не изменяет объект
    void eval_inplace(bool skip_simplify = false);     // заменяет дерево на CONST

    void simplify_inplace();                     // канонизация in-place
    LazyRational simplify() const;               // возвращает новую чистую копию

    internal::Interval approx_interval() const;  // кэшируемая приблизительная оценка

    void ensure_dirty();                         // приводит к грязному состоянию

    bool is_dirty() const;
    bool is_clean() const;

    // Мутирующие операторы (те же, что и ранее)
    friend LazyRational& operator+(LazyRational& a, const LazyRational& b);
    // ... и т.д.

    // Сравнения (многоуровневые)
    friend bool operator==(const LazyRational& a, const LazyRational& b);
    friend bool operator!=(const LazyRational& a, const LazyRational& b);
    friend bool operator<(const LazyRational& a, const LazyRational& b);
    // ...

private:
    enum class State { Dirty, Clean };
    mutable State state_;

    // Dirty:
    mutable std::vector<internal::DirtyNode> nodes_;
    mutable std::vector<internal::Value> constants_;
    mutable int root_ = -1;

    // Clean:
    mutable int clean_index_ = -1;

    // Кэш интервала:
    mutable std::optional<internal::Interval> cached_interval_;

    void canonicalize() const;                   // Dirty → Clean
    Rational eval_dirty() const;                 // вычисление грязного дерева без канонизации
    int import_tree(const LazyRational& other);
    int add_constant(const internal::Value& v);
    int new_dirty_node(internal::LazyOp op, absl::InlinedVector<int,2> children, int const_index = -1);
    void append_sum_children(int sum_node, const LazyRational& other);
    void append_product_children(int prod_node, const LazyRational& other);
    void invalidate_interval() const { cached_interval_.reset(); }
};
```

### 2.3. Структуры узлов

**DirtyNode** (в `lazy_nodes.h`) – использует `absl::InlinedVector<int,2>` для хранения детей, что оптимизирует память и скорость для малых арностей:
```cpp
struct DirtyNode {
    LazyOp op;
    absl::InlinedVector<int,2> children;   // индексы в nodes_
    int const_index = -1;
};
```

**TempNode** (там же, используется при канонизации):
```cpp
struct TempNode {
    LazyOp op;
    std::vector<int> children;
    int value_idx = -1;
    int eps_idx = -1;
    uint64_t hash;
    Interval approx;
    int32_t depth;
};
```

**Чистый Node** (в `node_pool.h`) – без изменений.

## 3. Примеры использования

### Пример 1. Накопление через `+=`
```cpp
LazyRational acc;
for (int i = 1; i <= 100000; ++i)
    acc += Rational(i);
Rational sum = acc.eval();            // канонизация + вычисление
```

### Пример 2. Быстрое вычисление без упрощения (гармонический ряд)
```cpp
LazyRational acc;
for (int i = 1; i <= 50000; ++i)
    acc += Rational(1, i);
Rational sum = acc.eval(true);        // пропуск канонизации, прямое вычисление
```

### Пример 3. Многоуровневое сравнение
```cpp
LazyRational a = ...;
LazyRational b = ...;
if (a == b) { ... }   // сначала интервалы, потом канонизация, при необходимости вычисление
```

Остальные примеры аналогичны предыдущей версии.

## 4. Инварианты (проверяемые контракты)

1. **`Rational`** никогда не содержит ленивого состояния.
2. **`LazyRational`** никогда не содержит immediate-значения.
3. Грязное дерево не имеет глобальных побочных эффектов.
4. После канонизации: `state_ == Clean`, грязные векторы пусты, `clean_index_` ссылается на существующий узел с `refcount > 0`.
5. Мутировать можно только грязное дерево (чистое сначала превращается в грязное через `ensure_dirty()`).
6. Операторы `+`, `-`, `*`, `/` мутируют левый операнд.
7. Узлы `SUB` и `DIV` не создаются.
8. В чистом дереве все `SUM`/`PRODUCT` каноничны (дети отсортированы, нет нейтральных элементов).
9. Все обходы итеративны, без рекурсии.
10. `approx_interval()` не вызывает канонизацию, кэширует результат.
11. `eval(true)` на грязном дереве не изменяет объект и не канонизирует.
12. Операторы сравнения используют многоуровневую логику.

## 5. Производительность и память

- Построение грязного дерева – амортизированное O(1) на операцию.
- Канонизация – O(N log N) из-за сортировки детей.
- Вычисление суммы через `eval(true)` – O(N) последовательных сложений с нормализацией, без экспоненциального роста знаменателя.
- Память грязного дерева: `sizeof(DirtyNode)*N + ...`.
- Глобальный пул чистых узлов использует хэш-таблицы для интернирования.
- Кэш интервалов – один `optional<Interval>` (16 байт).

```markdown
## 8. Перспективы: ленивые вычисления в матрицах и тензорах

### 8.1. Постановка задачи

После стабилизации ядра библиотеки (Rational + LazyRational) возникает естественный вопрос:
как эффективно использовать ленивые выражения в многомерных контейнерах (матрицы, векторы,
тензоры) для задач линейной алгебры, дифференциальной геометрии и символьных вычислений?

Архитектура версии 2.0 предлагает два различных пути, каждый со своими компромиссами.

### 8.2. Вариант 1: Холистический ленивый контейнер (`LazyMatrix` / `LazyTensor`)

Контейнер владеет **единым грязным деревом** для всех своих элементов. Элементы представлены
индексами внутри этого общего дерева. Операции над контейнерами мутируют общее грязное дерево.

**Преимущества:**
- **Максимальная производительность построения выражений:** все промежуточные операции происходят
  в одном графе, без лишних аллокаций на каждый элемент.
- **Единый кэш интервалов и упрощений:** канонизация выполняется один раз для всего контейнера.
- **Полный контроль над памятью и вычислительным графом.**
- **Естественная поддержка символьных переменных и производных на уровне контейнера.**

**Недостатки:**
- **Необходимость реализации всей линейной алгебры с нуля:** умножение, LU-разложение,
  собственные значения, решение систем и т.д. Это требует человеко-лет разработки и
  тестирования, чтобы хотя бы приблизиться к функциональности и производительности
  существующих библиотек (Eigen, Blaze, Armadillo).
- **Высокий порог входа для пользователей,** привыкших к стандартным интерфейсам.

**Вердикт:** теоретически идеальный, но практически нереализуемый в одиночку вариант.
Может рассматриваться только в рамках крупного исследовательского проекта с командой
разработчиков.

### 8.3. Вариант 2: Легковесная обёртка для Eigen (`CleanLazyRational`)

Создаётся отдельный тип `CleanLazyRational`, который:
- Хранит **только индекс корня** в глобальном пуле чистых узлов (4 или 8 байт).
- Имеет **семантику значения** (copyable, assignable).
- Каждая операция создаёт новый чистый узел в глобальном пуле (интернирование через кэши).
- Полностью совместим с `Eigen::NumTraits`.

**Преимущества:**
- **Бесплатное использование всей мощи Eigen:** все алгоритмы, SSE/AVX-оптимизации,
  выражения-шаблоны, решатели, геометрические трансформации — работают из коробки.
- **Минимальный объём нового кода:** `CleanLazyRational` — это ~100-150 строк,
  реализующих конструкторы, операторы и специализацию `Eigen::NumTraits`.
- **Автоматическое разделение подвыражений:** благодаря интернированию в `NodePool`,
  одинаковые подвыражения в матрице будут представлены одним узлом, экономя память
  и ускоряя вычисления.
- **Понятная ментальная модель:** `Rational` — immediate число, `CleanLazyRational` —
  ленивое число, которое можно копировать как `int`.

**Недостатки:**
- **Каждый элемент матрицы — отдельное дерево.** Это создаёт накладные расходы на
  хранение индексов и работу счётчиков ссылок (однако, благодаря интернированию,
  количество узлов растёт не линейно с числом элементов).
- **Невозможность мутаций на месте** (как в грязном дереве). Для построения больших
  выражений внутри циклов рекомендуется предварительно использовать `LazyRational`,
  а затем преобразовывать в `CleanLazyRational` через `simplify()`.

**Почему этот вариант логичнее:**

1. **Повторное использование проверенной инфраструктуры.** Eigen — стандарт де-факто
   в научных вычислениях на C++, его оптимизациям доверяют десятилетия.
2. **Быстрый выход в production.** Реализация займёт дни, а не годы.
3. **Гибкость.** Пользователь может смешивать `Rational`, `LazyRational` и
   `CleanLazyRational` в зависимости от этапа вычислений:
   - Построение модели: `LazyRational` с мутациями.
   - Формирование матрицы: преобразование в `CleanLazyRational` и заполнение Eigen-матрицы.
   - Решение системы: вызов `Eigen::solve()`.
4. **Естественная эволюция.** Если в будущем потребуется более глубокая интеграция,
   можно будет разработать специализированные адаптеры для Eigen, не ломая базовую
   архитектуру.

### 8.4. Набросок интерфейса `CleanLazyRational`

namespace delta {
    class CleanLazyRational {
        int root_index_ = -1;
    public:
        // Конструкторы
        CleanLazyRational();
        explicit CleanLazyRational(const Rational& r);
        CleanLazyRational(const LazyRational& lr); // требует, чтобы lr был Clean или канонизирует

        // Копирование и перемещение
        CleanLazyRational(const CleanLazyRational&);
        CleanLazyRational& operator=(const CleanLazyRational&);
        ~CleanLazyRational();

        // Арифметические операторы (создают новые узлы в пуле)
        friend CleanLazyRational operator+(const CleanLazyRational&, const CleanLazyRational&);
        friend CleanLazyRational operator-(const CleanLazyRational&, const CleanLazyRational&);
        friend CleanLazyRational operator*(const CleanLazyRational&, const CleanLazyRational&);
        friend CleanLazyRational operator/(const CleanLazyRational&, const CleanLazyRational&);
        friend CleanLazyRational operator-(const CleanLazyRational&);

        // Сравнения
        friend bool operator==(const CleanLazyRational&, const CleanLazyRational&);
        // ...

        // Вычисление
        Rational eval() const;

        // Доступ к индексу (для внутреннего использования)
        int root_index() const { return root_index_; }
    };
}

    };

### 8.5. Рекомендация

На данном этапе развития библиотеки **приоритетным направлением является реализация
`CleanLazyRational` и его интеграция с Eigen**. Это даст максимальный практический
эффект при минимальных затратах на разработку и позволит использовать библиотеку
в реальных научных и инженерных расчётах уже в ближайшее время.

Холистический `LazyMatrix` остаётся долгосрочной исследовательской целью, которая
может быть пересмотрена при появлении соответствующих ресурсов.
