# Δ‑analysis Roadmap: From Validation to Generation

**Status:** long‑term development plan, extending the library toward a
complete cycle of working with analytical and discrete representations
of functions.

---

## 1. Current Foundation (v0.2)

The library already supports, for a given analytical function and grid:

- Computing function values exactly (rational arithmetic) or with controlled
  tolerance (transcendental functions);
- Evaluating continuity, differentiability, and convergence of Riemann sums;
- Adaptive grid refinement in regions where the function deviates from
  linearity;
- Constructing discrete differential operators (DEC, finite differences) and
  verifying exact algebraic invariants.

---

## 2. Stage I: Symbolic Analysis of Functions (v0.3–v0.4)

Extension of `LazyRational` → `Expression<Scalar>` and construction of
`AnalyticalFunction`:

- Automatic detection of even/odd parity, periodicity;
- Localisation of extrema, zeros, asymptotes, and discontinuities;
- Classification of functions by analytical properties;
- Using these properties to **optimise grid construction** (see the dedicated
  backlog entry).

Result: grids built not “blindly”, but by consciously exploiting analytical
invariants.

---

## 3. Stage II: Inverse Problem — the Conversion Validator (v0.5–v0.6)

**Idea:** the library becomes a tool that, given discrete data (grid + values
+ known noise level), identifies the closest analytical function.

### How it works

1. Input: a dataset consisting of a sequence of converging numerical grids
   (e.g. one‑dimensional function values at grid points).
2. The noise level is known (e.g. ~1e-12); outliers may be present.
3. The library builds a uniform grid, computes discrete invariants (deviation
   from linearity, periodicity, symmetry, variational characteristics).
4. It compares the observed invariants against a catalogue of analytical
   function templates, accounting for noise and outliers.
5. It returns a **verdict**: “The data are generated by sin(x) with 99%
   confidence, given the noise level ~1e-12.”
6. It outputs an analytical expression (`Expression`) that can subsequently
   be used for exact computation.

### Key capabilities

- Robustness to noise and outliers;
- Based on the same analytical invariants that were proved for any
  discretisation (symmetry, monotonicity, convexity, periodicity);
- The result is not merely fitted coefficients, but a **proof‑backed
  conclusion** with a confidence estimate.

---

## 4. Symbiosis of Forward and Inverse Processes

The two directions form a closed loop:

- **Forward process** (analytical function → grid) uses proven continuous
  properties to construct an optimal discretisation.
- **Inverse process** (data → analytical function) uses the same invariants
  to recover the function that produced the data.

Thus, a single set of analytical invariants serves both for **generating
exact grids** and for **validating observed data**.

---

## 5. Further Horizons

- Multidimensional validator (2D/3D scalar and vector fields);
- Recovery not only of scalar functions but also of the differential
  operators they satisfy;
- Construction of a minimal analytical model explaining the data (Occam’s
  razor in action);
- Applications in experimental data processing, automated testing of
  numerical schemes, and anomaly detection.

---

## 6. Comment

Δ‑analysis was designed not merely to compute, but to **understand** what it
is computing. The introduction of symbolic analysis closes this circle: the
library begins not only to exploit mathematical properties of functions, but
also to infer those properties from data autonomously.

> *“We don’t just build grids. We prove where they came from.”*

## Why Sequence Beats a Flat Dataset

Conceptually it is the same data. A large grid contains a small one inside
it. Yet algorithmically they are worlds apart. Here is why analysing a
**sequence of progressively refined grids** dominates analysing a single
flat dataset.

### 1. Error hierarchy — the Matryoshka principle

In a flat dataset all points are equal before the law. A point at 1/2 and a
point at 1/4 are just two entries in a system of equations.

In Δ‑analysis:
- The point at 1/2 (from grid S₁) is a *base‑level* anchor.
- The points at 1/4 and 3/4 (from S₂) are *refinements*.

This allows the validator to compute the **local convergence rate**. If the
error on [0, 1/2] falls off as h² while on [1/2, 1] it falls off as h¹, the
validator concludes immediately: *“The right half contains either a kink
(non‑smoothness) or hidden noise.”*

A classical fitter would simply return the average temperature across the
whole hospital ward.

### 2. Dynamic noise filtering

Suppose the data carry noise ~1e-12:

- On grid S₁ (step 0.5) the discretisation error dwarfs the noise.
- On grid S₅ (fine step) the discretisation error has shrunk *below* the noise.
- The validator **sees this crossover**: up to some level accuracy keeps
  improving; beyond that it starts wandering. The physical precision limit
  of the data is now known.
- A flat dataset for a classical algorithm is merely a noisy cloud. The
  algorithm cannot tell where mathematics stops and garbage begins.

### 3. Computational economy — early stopping

- Flat dataset: you must crunch all N million points.
- Sequence: you analyse S₁, S₂, S₃. If at S₃ you already know with 99.9%
  confidence that the function is sin(x), you discard the remaining data.
  You do not need to process S₁₀ — the pattern is cracked.

This is the validator in action: seeking the **minimally sufficient
complexity** that explains the truth.

### 4. Structural integrity — the “betweenness” factor

The “betweenness” comparator operates naturally on a sequence. You ask:

*“I inserted point C between A and B. Did my prediction improve by a factor
of 2^k?”*

If the answer is *no*, the data structure has been violated — an outlier or
chaos has appeared. This check is only possible when you track the same
interval across multiple refinement levels.

### Example: the needle peak

Consider a function with a very narrow spike.

1. **Classical fitter on a flat dataset:** sees many zeros and one or two odd
   values. It treats them as outliers and smooths the spike away.
2. **Δ‑analysis validator on a sequence:** S₁ shows no peak. S₂ shows
   *something*. S₃ shows that *something* has grown exactly according to the
   grid refinement law.
3. **Verdict:** This is not noise. This is a real physical process, just
   highly localised.

### Summary

> Analysing a flat dataset is archaeology — sifting through a heap of bones.  
> Analysing a refinement sequence is embryology — watching how a full organism
> develops from a single cell.

In embryology you see the causal connection between resolution levels. That
connection is what allows you to separate *“law of nature”* from *“dirt on
the lens”*. Δ‑analysis is built to exploit exactly that connection.

### 2.1 Piecewise Analytical Segmentation

More often than not, real data are **not** generated by a single analytical
function across the whole domain. Phenomena change, sensors fail, regimes
switch.  The validator therefore operates on sub‑intervals and assembles a
**piecewise analytical model** automatically.

#### How it works

1. The domain is recursively partitioned into sub‑intervals.
2. On each sub‑interval the validator tests candidate analytical forms
   (`sin`, `exp`, polynomials, …) and computes a confidence score.
3. Where no simple analytical form meets the requested confidence, the
   sub‑interval is either refined further or a high‑degree polynomial
   surrogate is used.
4. The user specifies a **preset** (e.g. “prefer speed”, “prefer precision”,
   “prefer interpretability”), and the library chooses the optimal
   piecewise representation by trading off:
   - computational cost of evaluating the representation;
   - storage size;
   - accuracy vs. the raw data;
   - interpretability (analytical forms rank higher than generic polynomials).

#### Example

Imagine data on the interval [0, 15], collected with a known noise floor of
1e-12.  The validator analyses the sequence of grids on the whole domain and
then splits it recursively:

- Sub‑interval [0, 10]: the invariants converge exactly like `sin(x)`; the
  validator reports *“90% confidence that this is sin(x).  10% of the
  residual is not explained by measurement noise alone — 5% is noise, the
  remaining 5% is a genuine deviation from a pure sine wave.”*
- Sub‑interval [10, 15]: the behaviour is statistically different and
  resembles a completely different analytical form (e.g. an exponential
  decay).

Based on the user’s preset, the library then makes a decision without
bothering the user:

- **“Prefer speed”**: the whole [0, 10] is stored as `sin(x)`, sacrificing
  that 5% deviation in exchange for a 300% speed‑up in evaluation.
- **“Prefer precision”**: the sub‑interval is kept as a high‑degree
  polynomial that reproduces the data exactly, at the cost of evaluation
  time.
- **“Prefer interpretability”**: the validator highlights the breakpoint at
  x=10 as a statistically significant change, flags it as a candidate for a
  physical regime switch or a sensor fault, and prompts the user to
  investigate.

The result is a `PiecewiseExpression`:

[0,  10] → sin(x)      (confidence 90%, speed‑up 300% vs raw data)
[10, 15] → 0.8·e^{-x}  (confidence 85%, residual matches noise model)


#### Significance

This feature turns Δ‑analysis from a passive calculator into an **active
research assistant**.  It does not merely fit data — it highlights *where*
and *why* the data stop obeying a simple law, prompting the scientist to ask
the next question.  Breakpoints become candidates for bifurcation points,
phase transitions, or instrument faults — all backed by exact invariants
and transparent confidence estimates.
