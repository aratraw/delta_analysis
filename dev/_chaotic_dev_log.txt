## УРОКИ И ОСМЫСЛЕНИЕ ОШИБОК (АКТУАЛИЗИРОВАННЫЕ ПОД ТЕКУЩУЮ АРХИТЕКТУРУ)

---

### УРОК 1: НИКОГДА НЕ ВЫЗЫВАЙ GC В СЕРЕДИНЕ КАНОНИЗАЦИИ

**Что случилось:** 
В `canonicalize()` при создании чистых узлов и заполнении пула до порога `gc_threshold` вызывался `collect_garbage()`. GC видел частично инициализированный узел (уже с `op = CONST`, но `value_idx = -1`) и при попытке вычислить его значение обращался к `pool.values[-1]` → крах.

**Почему это ошибка:**
- Нарушение инварианта: узел не должен быть доступен для GC до полной инициализации
- GC не должен запускаться в момент, когда структуры данных находятся в несогласованном состоянии

**Как сделали правильно:**
- Ввели `thread_local bool gc_disabled`
- RAII-охранник `CanonicalizeGuard` устанавливает `gc_disabled = true` перед канонизацией
- В `allocate_node` проверка: `if (!gc_disabled && next_free_index >= gc_threshold)`
- Временно снимаем лимит `max_size = SIZE_MAX`, чтобы не провоцировать GC при расширении

**Почему это правильно:**
- Гарантирует, что GC не прервёт построение дерева
- После канонизации GC можно вызвать явно или он сработает при следующем выделении
- RAII обеспечивает восстановление даже при исключениях

---

### УРОК 2: GC ДОЛЖЕН СОХРАНЯТЬ ТОЛЬКО КОРНИ, А НЕ ВСЕ ЖИВЫЕ УЗЛЫ

**Что случилось:**
Старая версия `collect_garbage()` копировала все узлы с `refcount > 0` в новый пул, превращая каждый в константу. В результате после GC в новом пуле оказывалось столько же узлов, сколько в исходном дереве (~400), хотя живых корней было всего 3 (`root1`, `root2`, `root3`).

**Почему это ошибка:**
- Нарушение ожиданий: GC должен сжимать дерево, а не сохранять промежуточную структуру
- Размер пула после GC оставался большим, хотя должен был уменьшиться до количества корней
- `max_size` не имел смысла — пул жил своей жизнью

**Как сделали правильно:**
- GC использует `get_clean_objects_snapshot()` для получения списка корневых объектов
- Создаёт новый пул с размером `max_root_index + 1`
- Создаёт константы только для корневых индексов
- Все остальные узлы (промежуточные) отбрасываются

**Почему это правильно:**
- После GC в пуле остаётся ровно столько узлов, сколько чистых `LazyRational`
- `max_size` наконец-то обретает смысл — это лимит на количество узлов
- Память эффективно освобождается

---

### УРОК 3: НУЖЕН РЕЕСТР ЧИСТЫХ ОБЪЕКТОВ, НЕ ЭПОХИ

**Что случилось:**
Мы долго спорили между эпохами (добавить поле `clean_epoch_` в каждый объект) и реестром (хранить указатели на чистые объекты). Эпохи отвергли.

**Почему эпохи — плохое решение:**
- Добавляют 8 байт каждому объекту, хотя 99% объектов никогда не используют `reset_pool`
- Требуют проверки при каждом обращении к чистому состоянию
- Не решают проблему `decrement_ref` после уничтожения пула

**Как сделали правильно:**
- Реестр: `thread_local unordered_set<LazyRational*> g_clean_rationals`
- Регистрация только в `canonicalize()` (когда объект становится чистым)
- Дерегистрация в `ensure_dirty()`, деструкторе, конструкторе перемещения
- `reset_pool()` обходит снимок реестра и инвалидирует объекты

**Почему это правильно:**
- Не добавляет полей к объекту
- Полный контроль над всеми чистыми объектами
- `reset_pool` и `collect_garbage` могут работать с реестром напрямую

### УРОК 4: ХЕШ-ФУНКЦИИ ДОЛЖНЫ БЫТЬ ИДЕАЛЬНО СОГЛАСОВАНЫ С РАВЕНСТВОМ

**Что случилось:**
`absl::flat_hash_map` крашился, потому что `operator==` говорил, что два `Value` равны (после нормализации), а `hash` возвращал разные значения. Это происходило, потому что хеш считался с неканонического представления.

**Почему это ошибка:**
- `absl::flat_hash_map` требует, чтобы `hash(a) == hash(b)` для всех `a == b`
- Иначе контейнер не может найти ключ и впадает в бесконечный цикл или assertion

**Как сделали правильно:**
```cpp
template <typename H>
H AbslHashValue(H h, const Value& v) {
    const auto& n = numerator(v);
    const auto& d = denominator(v);
    return H::combine(std::move(h), n, d);
}
```
В `Value` нет флага "редуцированности" — дробь всегда хранится в нормализованном виде благодаря `rational_adaptor`. Поэтому хеш можно считать напрямую с числителя и знаменателя.

**Почему это правильно:**
- Нормализация гарантируется типом на уровне бэкенда
- Не нужно отдельно нормализовать перед хешированием
- Устраняет класс ошибок, связанных с неканоническим представлением

### УРОК 5: ПРИ КОМПОЗИЦИИ ФУНКЦИЙ ПЕРЕДАВАТЬ СТРОГИЙ EPSILON В ПОДВЫЧИСЛЕНИЯ

**Что случилось:**
В `eager_pow(base, exp, eps)` для рационального показателя вызывались `eager_log(base, eps)` и `eager_exp(... , eps)`. Ошибка от лога и экспоненты накапливалась, и итоговая погрешность могла превысить `eps`.

**Почему это ошибка:**
- Пользователь просит точность `eps`, а получает `n * eps` (где `n` — число подвычислений)
- Особенно критично для больших показателей степени

**Как сделали правильно:**
```cpp
Value internal_eps = (p == 0) ? eps : eps / Value(p * 1000);
Value log_base = eager_log(base, internal_eps);
// ...
Value result = eager_exp(p_log_div_q, internal_eps);
```

**Почему это правильно:**
- Запас 1000 обеспечивает, что итоговая ошибка будет доминироваться последним шагом
- Пользователь не видит разницы, но библиотека надёжна

### УРОК 6: `reset_pool()` ДОЛЖЕН ИНВАЛИДИРОВАТЬ ЧИСТЫЕ ОБЪЕКТЫ ЧЕРЕЗ PLACEMENT NEW

**Что случилось:**
После `reset_pool()` старый пул уничтожался, но чистые `LazyRational` продолжали хранить `clean_index_`, указывающий в уничтоженную память. При следующем использовании этих объектов происходил краш.

**Почему это ошибка:**
- Инвариант: чистое состояние означает валидный индекс в пуле
- `reset_pool()` нарушает этот инвариант, не уведомляя объекты

**Как сделали правильно:**
```cpp
auto clean_objects = get_clean_objects_snapshot();
for (LazyRational* obj : clean_objects) {
    decrement_ref(obj->clean_index_);
    obj->~LazyRational();
    new (obj) LazyRational();   // грязный ноль, не регистрируется
}
pool = NodePool();
clear_clean_registry();
```

**Почему это правильно:**
- Объект становится грязным нулём — начальное состояние по умолчанию
- Счётчики ссылок старого пула корректно уменьшены до уничтожения пула
- Реестр очищается, так как грязные объекты не должны в нём находиться

### УРОК 7: `clone()` ЧИСТОГО ОБЪЕКТА ДОЛЖЕН РЕГИСТРИРОВАТЬ КОПИЮ

**Что случилось:**
`clone()` для чистого объекта создавал копию с тем же `clean_index_`, увеличивал счётчик ссылок, но НЕ добавлял копию в реестр. При `reset_pool()` эта копия не находилась в реестре и не инвалидировалась, оставаясь с висячим индексом.

**Почему это ошибка:**
- Инвариант: каждый чистый объект должен быть зарегистрирован в реестре
- Клонирование создаёт новый чистый объект, который обязан быть в реестре

**Как сделали правильно:**
```cpp
LazyRational copy;
copy.state_ = State::Clean;
copy.clean_index_ = clean_index_;
internal::increment_ref(clean_index_);
copy.register_clean();   // добавлено
return copy;
```

**Почему это правильно:**
- Поддерживается инвариант "все чистые объекты в реестре"
- `reset_pool()` и `collect_garbage()` видят все копии

### УРОК 8: НЕ ДОВЕРЯЙ `to_double()` ПРИ ОТЛАДКЕ ТРАНСЦЕНДЕНТНЫХ ФУНКЦИЙ

**Что случилось:**
Часами смотрели на вывод `EXPECT_LE`, на конвертацию в double, но не видели разницы. Оказалось, что `to_double()` терял точность, скрывая истинную разницу между ожидаемым и фактическим значениями.

**Почему это ошибка:**
- `double` имеет только 15-17 десятичных цифр точности
- Рациональные числа могут иметь сотни цифр
- Сравнение через `to_double()` не показывает настоящей погрешности

**Как сделали правильно:**
```cpp
std::cout << "val = " << internal::to_string(val) << "\n";
std::cout << "expected = " << internal::to_string(expected) << "\n";
std::cout << "diff = " << internal::to_string(diff) << "\n";
```

**Почему это правильно:**
- Видна полная точность рационального числа
- Видно, сколько знаков на самом деле не совпадает
- Экономит часы отладки

---

### УРОК 9: ИЗБЕГАЙТЕ ПРОИЗВОЛЬНЫХ ОГРАНИЧЕНИЙ НА КОЛИЧЕСТВО ИТЕРАЦИЙ В РЯДАХ

**Что случилось:**
Некоторые ряды (например, `series_ln2`) имели `max_iter = 10000` и останавливались по достижении лимита, а не по достижению точности. Для очень малых `eps` (1e-100) точность не достигалась.

**Почему это ошибка:**
- Пользователь запросил точность `eps`, но получил гораздо худшую
- Нарушение контракта функции

**Как сделали правильно:**
- `DEFAULT_MAX_ITER = 1'000'000` только как защита от расходимости
- Основное условие остановки: `term < eps`
- Пользовательский `eps` имеет приоритет перед любым лимитом

**Почему это правильно:**
- Если ряд сходится, он будет идти до нужной точности
- Если ряд расходится, защита сработает и не зависнет

### УРОК 10: ПРИ ВНЕСЕНИИ ИЗМЕНЕНИЙ В API — GREP ВСЁ, ПОТОМ GREP ЕЩЁ РАЗ

**Что случилось:**
При изменении API (например, добавлении `register_clean`) забыли обновить `clone()` для чистого объекта, что привело к багу.

**Почему это ошибка:**
- API изменения имеют побочные эффекты во всех местах использования
- Легко пропустить, особенно если изменения "очевидны"

**Как сделали правильно:**
- После изменения API — систематический поиск всех мест использования
- Перекомпиляция и просмотр всех ошибок
- Двойная проверка: если что-то должно регистрироваться — везде, где создаётся чистый объект

**Почему это правильно:**
- Предотвращает забытые обновления
- Экономит время на отладку

### УРОК 11: КОД ДОЛЖЕН БЫТЬ «ГЛУБОКИМ», А НЕ «ШИРОКИМ». ПЛОТНОСТЬ СМЫСЛА — ЭТО ХОРОШО

**Что случилось:**
Мы не стали плодить абстракции ради абстракций. Выбрали `thread_local` везде, где можно. Не добавляли лишние поля. Не делали "эпохи" через поле в каждом объекте.

**Почему это правильно:**
- Высокая плотность функционала на строку кода
- Меньше мест для ошибок
- Легче поддерживать инварианты

## ИТОГ ТЕКУЩЕЙ АРХИТЕКТУРЫ (что должны помнить)

1. **Никогда не вызывать GC в канонизации** — `gc_disabled` + RAII охранник
2. **GC сохраняет только корни** — реестр чистых объектов говорит, что живое
3. **Реестр чище, чем эпохи** — не раздувает объекты
4. **Хеш всегда от нормализованного представления** — благодаря `rational_adaptor`
5. **Композиция функций требует более строгого eps в подвычислениях** — запас 1000
6. **`to_string()` для отладки, а не `to_double()`** — вся точность видна
7. **Клонирование чистого объекта = регистрация копии** — инвариант реестра
8. **`reset_pool` инвалидирует через placement new** — грязный ноль, дерегистрация
9. **Итерационные ряды не ограничены произвольными лимитами** — правит `eps`
10. **API изменения требуют систематического поиска** — grep, компиляция, grep


##[26.04] БАГ-РЕПОРТ: `vector subscript out of range` в тесте `GCAndResetInteraction`

### Условия возникновения

1. Установлен лимит пула `max_size = 120`
2. Создаётся сложное ленивое выражение `acc` как сумма 80 термов `Sin(i) * Cos(i+1)`
3. Вызывается `acc.simplify_inplace()` (канонизация грязного дерева в чистое)
4. В процессе канонизации создаются сотни новых узлов в пуле
5. При достижении `next_free_index >= gc_threshold` (108) срабатывает **рекурсивный вызов** `collect_garbage()`
6. GC начинает вычислять значение каждого живого узла, включая **только что выделенный, но ещё не полностью инициализированный** узел типа `CONST` с `value_idx = -1`
7. При попытке доступа к `pool.values[-1]` происходит `vector subscript out of range`

### Корень проблемы

**Рекурсивный вызов GC внутри `canonicalize`** — в момент, когда пул заполнен, а канонизация ещё не завершена. Узлы создаются в таком порядке:
1. `allocate_node()` выделяет слот и возвращает индекс
2. **До присвоения `value_idx`** (или до полного заполнения полей узла) проверяется порог и вызывается GC
3. GC видит узел с `op = CONST` (уже установлен конструктором `Node`), но `value_idx = -1` (ещё не присвоен)
4. `evaluate()` пытается прочитать `pool.values[-1]` → крах

### Дополнительные факторы

- `max_size` слишком мал (120) относительно сложности выражения (требуется несколько сотен узлов)
- `gc_threshold = 0.9 * max_size = 108` — порог достигается задолго до завершения канонизации
- Отсутствие механизма, запрещающего GC в критические моменты

### Решение (комплексное)

1. **Введён флаг `gc_disabled`** (thread-local) в `global_state.h`
2. **RAII-охранник `CanonicalizeGuard`** в `lazy_rational_impl.h`:
   - Устанавливает `gc_disabled = true`
   - Временно снимает лимит `max_size = SIZE_MAX` (через `pool.max_size = SIZE_MAX`)
   - В деструкторе восстанавливает исходные значения
3. **Модифицирован `allocate_node`** в `node_pool.h`:
   - GC запускается только если `!gc_disabled && next_free_index >= gc_threshold`
   - При отключённом GC разрешается расширять пул без ограничений
4. **Переписан `collect_garbage`**:
   - Использует реестр чистых объектов (`g_clean_rationals`) для определения корней
   - Создаёт новый пул размером `max_root_index + 1`
   - Сохраняет только корневые узлы как `CONST`
   - Не копирует все живые узлы, а только корни
5. **Введён реестр чистых объектов** `g_clean_rationals`:
   - Регистрация в `canonicalize()` после перехода в `Clean`
   - Дерегистрация в `ensure_dirty()`, деструкторе, конструкторе перемещения
   - Используется в `reset_pool()` и `collect_garbage()`
6. **Исправлен `clone()` для чистого объекта** — добавлен вызов `register_clean()`
7. **Исправлен `reset_pool()`** — обходит реестр, инвалидирует чистые объекты через `placement new`

### Купирование (воспроизведение и фиксация)

- **Воспроизведение**: тест `GCAndResetInteraction` из `reset_pool_edge_cases_tests.h`
- **Фиксация**: все вышеперечисленные изменения, проверенные через verbose-тесты

### Мораль

1. **Никогда не вызывай GC в середине построения сложной структуры данных** — используй флаг отключения.
2. **Реестр объектов** проще и безопаснее эпох: не добавляет полей, даёт полный контроль.
3. **RAII — лучший друг**: охранник гарантирует восстановление состояния даже при исключениях.
4. **Тесты должны быть агрессивными**: 300 итераций с созданием/уничтожением объектов выявили проблему, которая при «оптимальном» коде могла бы остаться незамеченной.
5. **Архитектура должна поддерживать инварианты**: пул не должен содержать узлов с `value_idx = -1`, а GC не должен запускаться до завершения инициализации.

### P.S.

Баг был многослойным, но системный подход (реестр + отключение GC + переписанный GC) решил проблему окончательно. Джобс был бы доволен — мы вытеснили пузыри и теперь можем вставить ещё одну микросхему.

### УРОК 12: ИЗОЛИРОВАННЫЙ ТЕСТ ПРОХОДИТ, В ОБЩЕМ ПРОГОНЕ ВИСНЕТ — ПРИЗНАК ГЛОБАЛЬНОГО ЗАГРЯЗНЕНИЯ

**Симптом:**  
Тест `EagerPowRationalExponent` (eager-вычисления степени с рациональным показателем) зависает при запуске после сотни других тестов, но успешно завершается, если запустить только сюиту `RationalPowTest`.

**Анализ:**  
Зависание не воспроизводится изолированно → проблема не в логике самого `eager_pow`, а в межтестовом взаимодействии. После выполнения десятков других тестов глобальное состояние оказывается «загрязнённым».

**Основные подозреваемые:**
- `thread_local` реестр `g_clean_rationals`
- `pi_cache` или другие внутренние кэши
- **`default_eps`** — глобальное значение точности по умолчанию
- Пул узлов

**Корень проблемы:**  
Глобальное состояние в библиотеке продумано и работает корректно (в целом). Проблема в тестах, которые не изолируют свои побочные эффекты друг от друга. Если бы разработка велась разными людьми, можно было бы сказать: *«Ну идите сами нормально настраивайте и изолируйте свои тестовые сьюты»*. Но на проекте один разработчик, и орать не на кого.

**Почему проблема трудноуловима:**
- Зависание/тормоза воспроизводятся только при определённом порядке выполнения тестов.
- В изолированной среде глобальное состояние девственно чистое → тест пролетает за миллисекунды.
- Разные тесты по-разному влияют на глобальное состояние.

**Выводы:**
- Глобальное состояние — не зло, это инструмент. Как нож: можно колбасу порезать, а можно человека убить (но лучше не нужно).
- В библиотеке глобальное состояние продумано на 90% идеально (по мнению разработчика). Проблема не в нём, а в тестовом фреймворке/культуре, где тесты не обязаны изолировать друг от друга свои изменения глобальных настроек.
- Наличие `set_default_eps()` подразумевает наличие `reset_default_eps()` — для симметрии и удобства тестирования.

**Решение (принятое в текущей версии):**
- Добавлен `reset_default_eps()` для возврата значения точности к исходному (`1e-30`).
- В тесты, чувствительные к глобальному состоянию, добавлен сброс в `SetUp`/`TearDown`.
- `reset_pool()` **не** сбрасывает `default_eps` (это разные семантические сущности: пул — управление памятью, `default_eps` — политика вычислений). Сброс политики — ответственность тестов.

### P.P.S. РАСПУТЫВАЯ МАТРЁШКУ (ПОЛНАЯ ВЕРСИЯ ДЕТЕКТИВА)

История с `RepeatingSubgraphInterning` — классический пример того, как **один симптом, а причин — несколько, и они лежат на разных уровнях**.

**Что мы видели:** тест зависает. Ни строчки вывода, ни намёка.

**Что мы заподозрили:** `reset_pool()`. Он и раньше был косвенно связан с зависаниями (уроки 1-11). В том бенчмарке ресеты пула вызывались на каждый чих — опасения были более чем обоснованными. Это был **educated guess**.

**Что сделали:** начали моделировать сценарии с `reset_pool()` — сделали отдельную тестовую сюиту.

**Хронология ада:**

1. Мы видим, что бенчмарк висит.
2. Главный подозреваемый — `reset_pool()`.
3. Начинаем моделировать сценарии с `reset_pool()` — пишем отдельную тестовую сюиту.
4. Тест `GCAndResetInteraction` из этой сюиты ловит краш (vector subscript out of range).
5. Это баг, его надо чинить. Чиним. Полтора дня.
6. Вкатываем изменения: реестр, `gc_disabled`, `CanonicalizeGuard`, placement new, переписанный GC.
7. Тесты на `reset_pool()` становятся зелёными. Архитектура упрочилась.
8. Возвращаемся к главному висящему бенчмарку `RepeatingSubgraphInterning`.
9. Он **всё ещё висит**, блядь.

Вот тут-то и начинается следующий акт: simplify, дистрибутивность, эвристики, группировка по хешам — думаем, что проблема в отсутствии схлопывания повторяющихся поддеревьев. И только потом — обнаружение `default_eps` и добавление `reset_default_eps()`.

**Что оказалось на самом деле:** виноватых было несколько, и они лежали на разных уровнях:

1. **`reset_pool()`** был виноват, но не в том, в чём мы думали. Мы создали сложный тестовый сценарий который сам себе сделал проблему и провалился, под этот провал мы чинили архитектуру.
2. **`default_eps`** был, вероятнее всего, изменён другим тестом на чудовищно маленькое значение (1e-100). После `reset_pool()` пул становился чистым, но `default_eps` оставался супер-маленьким.
3. **Отсутствие дистрибутивности и сворачивания повторяющихся поддеревьев** — simplify не мог достаточно сильно сократить дерево, и оно оставалось большим. Что, для трансцендентных членов, оказывает значительное влияние. Хорошее трансцендентное - это то, которое можно вынести за скобки. Лучшее трансцендентное - это то, которое не надо считать вовсе. 
4. **Супер-маленький `default_eps` + большое дерево трансцендентных, которое не сокращалось без обработки дистрибутивности** = вычисление требовало чудовищной точности и времени. Тест не висел — он **честно считал Истину до тепловой смерти Вселенной**.

**Итог:** 
- Тест `GCAndResetInteraction` был не причиной, а **случайной находкой** по дороге.
- Мы не зря чинили `reset_pool()` и GC — это правильные фичи, нужные библиотеке самой по себе.
- Мы не зря добавили дистрибутивность и сворачивание — это тоже правильные фичи.
- Но главный бенчмарк требовал всего этого **плюс** сброса точности.

**Главный урок:** 
- Educated guess — это хорошо. Но проверка гипотезы должна быть быстрой и дешёвой.
- Если у тебя висит тест — будь готов к тому, что по дороге ты найдёшь ещё три бага, два из которых окажутся не связанными с оригинальным симптомом, и только последний, четвёртый, окажется тем самым. А остальные ты просто починишь по пути, потому что ты теперь знаешь, как это делается.
- Прежде чем переписывать архитектуру под подозреваемого, удостоверься, что он действительно виноват. Или хотя бы что он — единственный виноватый.
